# libvaxis / vxfw (Zig) — the substrate bar

**Verdict:** libvaxis is the reference implementation for what a modern terminal substrate must do — query-based capability negotiation, negotiated (not hardcoded) grapheme width, in-band resize — and its vxfw layer is a cautionary tale that a great substrate does not automatically yield a great framework.

## Core architecture

**Three-primitive substrate.** A `Tty` (platform I/O), a `Vaxis` (screen state + capability state + renderer), and an event loop you can replace. `USAGE.md` states the entire event-loop contract in three lines: read raw bytes from the TTY, feed them to the parser, handle the returned events — plus one duty people miss: forward capability-response events back into `Vaxis` so it knows what it may use. A worked libxev integration is included (`USAGE.md:20-120`). The bundled `Loop(comptime T)` is generic over the *user's* event enum (`src/Loop.zig:13`) — library events are mixed into the app's own event type rather than wrapped. Input runs on a concurrent task (`src/Loop.zig:73`); `stop()` unblocks the blocking TTY read by sending a DSR query so the terminal itself generates wake-up bytes (`src/Loop.zig:85`) — a genuinely clever trick, no self-pipe needed.

**Capability negotiation, no terminfo.** `queryTerminal` (`src/Vaxis.zig:257`) writes one batched burst of queries (`queryTerminalSend`, `src/Vaxis.zig:278`): DECRQM for sgr-pixels/2027/2031, mode 2048 set, XTVERSION, kitty-keyboard CSI-?u, kitty graphics, and — the neat part — *behavioral* probes: home the cursor, emit an explicit-width or scaled-text sequence, then ask for the cursor position; if the column moved, the feature exists (`src/Vaxis.zig:300-316`). Everything is fenced by a trailing DA1: every terminal answers DA1, so its arrival means all prior answers are in. The caller blocks on a futex with a timeout (1s in vxfw) until DA1 wakes it. `enableDetectedFeatures` (`src/Vaxis.zig:329`) then turns things on, with environment escape hatches layered on top: `VHS_RECORD` forces wcwidth+no-kitty, `TERM_PROGRAM=vscode` forces legacy SGR, plus `VAXIS_FORCE_WCWIDTH`/`VAXIS_FORCE_UNICODE` user overrides (`src/Vaxis.zig:340-353`). Mode 2027 is only enabled if the terminal lacks the kitty explicit-width protocol (`src/Vaxis.zig:360`) — the newer protocol supersedes it.

**Width is negotiated, not assumed.** `src/gwidth.zig:7` defines three width methods — `unicode` (grapheme clusters, EAW, VS15/VS16, RI flag pairs), `wcwidth`, `no_zwj` — and the *active method is a capability*: `self.screen.width_method = self.caps.unicode` (`src/Vaxis.zig:203`). The renderer measures with the same method at draw time (`src/Vaxis.zig:459`). This is the core insight: width is a contract between program and terminal, and the program must adopt whichever segmentation the terminal actually uses, or cursor positions drift.

**Rendering.** Cells are `{ grapheme: []const u8, width: u8, style, link, scale }` (`src/Cell.zig`); multi-codepoint graphemes are interned in an 8 KiB wrap-around `GraphemeCache` (`src/GraphemeCache.zig:14` — note it just resets to 0 on overflow, silently aliasing old slices; a hazard, not a design to copy). `render` (`src/Vaxis.zig:375`) is a classic double-buffer diff against `screen_last`, wrapped in mode 2026 sync (`src/Vaxis.zig:424`) with an `errdefer` that resets sync on failure; wide-cell continuation cells get `skipped` flags (`src/Vaxis.zig:469`); it deliberately relies on terminal auto-wrap instead of repositioning at row ends, which is what makes non-alt-screen (inline) rendering work.

**vxfw.** Introduced 2024-10-30, ~9 months after the substrate, incubated out-of-tree first. Flutter-shaped: `Widget` is a hand-rolled vtable `{ userdata: *anyopaque, captureHandler?, eventHandler?, drawFn }` (`src/vxfw/vxfw.zig:273`). Each frame, `drawFn(DrawContext{arena, min, max, cell_size})` returns a `Surface` — an arena-allocated cell buffer plus positioned `SubSurface` children with z-indexes (`src/vxfw/vxfw.zig:338`). The runtime (`src/vxfw/App.zig:52`) runs a fixed-framerate loop (default 60fps, `App.zig:85`), drains the event queue, dispatches DOM-style — capture phase root→target, then bubble target→root (`App.zig:588-602`) — hit-tests mouse events against the *last rendered frame's* surface tree (`App.zig:363`), rebuilds `path_to_focused` from the new surface tree every frame (`App.zig:534`), and re-renders only when a handler set `ctx.redraw` or a `Command` (tick timer, request_focus, set_mouse_shape, query_color, notify) implies it. Widget identity is pointer equality: `userdata` ptr + `drawFn` ptr (`src/vxfw/vxfw.zig:296-299`). ~16 widgets ship in `src/vxfw/`: FlexRow/FlexColumn, ListView, ScrollView, TextField, RichText, SplitView, Spinner, Border, Padding, Center, SizedBox, Button…

## What it gets right

- **The negotiation burst.** One write, DA1 fence, futex+timeout, then enable. Cheap, race-free, degrades to a 1s timeout on dumb terminals. Behavioral cursor-position probes detect features that have no query form.
- **Progressive enhancement as ideology.** Kitty keyboard, 2026, 2027, 2031, 2048, OSC 8/52/9/777, kitty graphics, explicit width — each independently detected and independently used. No `TERM` string parsing anywhere.
- **Width method as negotiated capability** with env overrides for known liars (VHS, vscode terminal). Nobody else does this; ratatui hardcodes unicode-width and prays.
- **In-band resize preferred over SIGWINCH** — vxfw only installs the signal handler when mode 2048 is absent (`src/vxfw/App.zig:70-76`). The author literally wrote the spec because signals are racy.
- **Replaceable event loop with a tiny contract.** The substrate doesn't own your runtime; the libxev example proves it. The DSR-to-unblock-read trick avoids platform-specific wakeup machinery.
- **Frame-accurate mouse hit-testing** against the last-rendered surface tree — what the user clicked is what was on screen, not what the next layout would produce.
- **Capture→bubble event phases + commands.** Widgets return effects (`ctx.tick(ms, widget)`, `request_focus`) instead of mutating the runtime — trivially testable.
- **Arena-per-frame draw allocation** (`DrawContext.arena`): widgets never free; surfaces die wholesale after render.
- **Inline (non-alt-screen) rendering** is first-class, via auto-wrap reliance and `\r`+RI cursor repositioning (`src/Vaxis.zig:427-434`).

## What users complain about

- **Widget identity via raw pointers is a footgun.** Discussion #232: user builds `Text` widgets on the fly for a ListView; they immediately dangle ("dangling stack pointers") because vxfw retains `*anyopaque` into user stack frames. The community answer is "keep an ArrayList of widgets with stable pointers" — i.e., the user hand-implements retained state the framework should own. (https://github.com/rockorager/libvaxis/discussions/232)
- **Terminal left corrupted on crash.** Discussion #165: segfaults/leaks leave the terminal unresponsive; rockorager's fixes are workarounds — "redirect stderr to a file and `tail -f`", install `pub const panic = vaxis.panic_handler;`, add your own ctrl-c bind. (https://github.com/rockorager/libvaxis/discussions/165)
- **Unicode edge cases in shipped widgets.** Issue #171: typing any character after 'ä' in TextInput panicked — the substrate got graphemes right but the widget layer indexed by bytes. (https://github.com/rockorager/libvaxis/issues/171)
- **Layout arithmetic bugs.** Issue #201: integer overflow in FlexRow — hand-rolled u16 layout math without a real constraint solver. (https://github.com/rockorager/libvaxis/issues/201)
- **Zig-version and dependency churn dominates the tracker.** #297/#233/#138/#72/#40 are all "doesn't build on Zig X"; #1/#16 are unicode-data dependency migrations (ziglyph→zg→uucode, visible in git history); v0.5.1 shipped only because the zg URL 404'd after a maintainer handoff. Ecosystem instability, not design flaws — but users feel it as "libvaxis is broken this month."
- **Rendering regressions at the edges**: discussion #180 "Scrolling breaks the rendering" — diff-based renderers accumulate subtle invariants (skip flags, wrapped cells) that break in hard-to-bisect ways.
- **Thin docs**: no book/guide beyond README + USAGE.md + examples; discussions (#142 tick, #344 suspend, #345 centering) are the de-facto documentation.

## Testing story

(All observations from the local clone @ ca781b3; run via `zig build test`, `build.zig:113`.)

- **~140 inline Zig `test` blocks, concentrated where the pure logic is.** `src/Parser.zig:750` onward is a long suite of input-decoding tests ("parse: single xterm keypress", ctrl/alt/shift combos, kitty sequences); `src/gwidth.zig:152-212` tests all three width methods against the hard cases (ZWJ emoji, VS15/VS16, skin tones, flag pairs, keycap sequences, combining marks).
- **Widgets are headless-testable by construction, and the shipped tests prove it.** Because `Widget`, `EventContext`, and `DrawContext` are plain values, a test builds a widget, synthesizes mouse/key events, calls `handleEvent`, then draws into a per-test arena and asserts on the returned `Surface` — see `test Button` (`src/vxfw/Button.zig:122`): press/release/leave/re-enter sequences against `onClick` counts, then size/child-origin assertions on the drawn surface. Text asserts on individual `surface.buffer[...]` cells (`src/vxfw/Text.zig:502`); TextField has 17 tests, ListView 7. No terminal, no PTY.
- **A `TestTty` exists but is barely used.** `Tty` is swapped at compile time under `builtin.is_test` (`src/tty.zig:17-22`) for a pipe-backed fake (`src/tty.zig:787`) that captures output in an allocating writer and reports a fixed 80×40 winsize — the skeleton of a headless driver. But nothing in the repo drives a full `App` through it: the only `App.zig` test (`src/vxfw/App.zig:612`, timer-consume semantics) constructs the App with `.tty = undefined, .vx = undefined`, i.e. runtime logic is tested only where it never touches the terminal.
- **The render diff — the state machine that actually regresses (#180) — has one test.** `src/Vaxis.zig:1509` ("render: no output when no changes") renders into an in-memory writer and asserts zero bytes. No snapshot/golden facility, no buffer-diff assertions, no property tests over skip/wrap flags; every surface assertion in the repo is hand-written `expectEqual` on cells and sizes.

Net: good unit discipline at the leaves (parsing, width, widgets), a stubbed-but-unused integration seam in the middle, and near-zero coverage of the renderer and event-loop plumbing — which is exactly where users report breakage (#180 scrolling corruption, #165 crash cleanup).

## What's worth stealing

- **The whole negotiation design**: batched query burst → DA1 fence → timeout → enable, with behavioral cursor-position probes and per-feature booleans instead of a terminfo-style capability database.
- **Negotiated width method** threaded through cell writing, layout measurement, and the render diff as one value — plus env-var overrides for multiplexer/recorder/embedded-terminal liars.
- **Mode 2048 first, SIGWINCH fallback**, decided by the same negotiation.
- **DSR-as-wakeup** for unblocking a blocking TTY read without a self-pipe.
- **Event-loop-agnostic substrate contract** (bytes → parser → events → feed capability events back), documented with a real third-party-loop example.
- **Capture/bubble phases, command lists, frame-accurate hit-testing, per-frame arena** from vxfw — the runtime shape is right even where the identity model is wrong.
- **`vaxis.panic_handler`** — a substrate-provided panic/unwind hook that restores the terminal (kitty keyboard pop, main screen, cursor show, sync reset) is table stakes.
- **Inline-mode rendering** via auto-wrap + RI repositioning rather than alt-screen-only.

## Implications for rabbitui

- **qwertty must grow a query router and a negotiation sequence, and rabbitui should refuse to run without it.** qwertty's README already admits "does not include a general terminal query router yet" — that router (route DECRPM/DA1/XTVERSION/CSI-u/kitty-graphics responses to waiting futures, pass everything else through as input) is the single highest-value substrate feature libvaxis demonstrates. Do the DA1-fence + timeout pattern; it maps cleanly onto qwertty's async session owner as `session.negotiate().await`.
- **Make width method a runtime value on the render/layout context, not a compile-time choice of unicode-width crate.** Expose `wcwidth | unicode | no_zwj` (gwidth.zig's trichotomy), select it from mode-2027/explicit-width negotiation, honor override env vars. Every text-measurement call in layout must go through it — libvaxis proves layout and renderer disagreeing with the terminal is *the* class of TUI jank.
- **Do not use pointer identity for widgets.** vxfw's `userdata`-ptr identity is its worst wart (discussion #232) and it's exactly what Rust borrow rules make miserable anyway. Use stable keys/IDs in a framework-owned tree (Xilem/Masonry-style) so "list of ephemeral view values" — the thing that broke vxfw users — is the *default* usage, not a footgun.
- **Steal the two-phase (capture→bubble) dispatch and command-list effects** for the event system: widgets get `&mut EventCtx`, set `consume`/`redraw`, and push commands (timers, focus requests, terminal queries like `query_color`) rather than touching the runtime. This also gives a natural async seam: commands can resolve as futures.
- **Prefer in-band resize (2048) and treat SIGWINCH as fallback**, chosen at negotiation time exactly like vxfw's conditional handler install. rabbitui's async loop then has *zero* signal handling on modern terminals.
- **Render pipeline: double-buffer diff wrapped in mode-2026 sync, with wide-grapheme skip cells and auto-wrap reliance for inline mode.** But add invariant checks/property tests around the skip/wrap flags — libvaxis's #180-style regressions show this state machine needs tests, not care.
- **Ship a panic/drop guard that restores terminal state on any exit path** (Rust makes this easier than Zig: `Drop` + `std::panic::set_hook`). libvaxis users hitting hosed terminals (#165) is the most common first-hour failure; rabbitui can make it near-impossible.
- **Don't intern graphemes in a fixed ring buffer.** GraphemeCache's silent-overwrite-on-overflow (`src/GraphemeCache.zig:14`) is UB-adjacent; use `compact_str`/`Arc<str>`-style small-string cells instead.
- **Copy the widget-test ergonomics, but build the missing harness.** vxfw's plain-value `EventContext`/`DrawContext` make widget tests three lines of setup (`Button.zig:122`) — rabbitui's contexts should stay equally constructible without a runtime. But libvaxis stops there: its `TestTty` seam exists and goes unused, and the diff renderer has a single test while being the top source of user-visible regressions. rabbitui should ship what libvaxis stubbed — a headless full-app driver (feed synthetic bytes/events, render to an in-memory buffer) plus buffer snapshot assertions, so renderer invariants are tested at the layer where #180-class bugs actually live.
- **Built-in widgets must be grapheme-correct from day one** — vxfw's TextInput panic after 'ä' (#171) shows a correct substrate doesn't protect a byte-indexing widget. Make the only string APIs rabbitui widgets see be grapheme-cursor APIs.
- **Framework-after-substrate works — plan for it.** libvaxis shipped bare, incubated vxfw externally for months, then absorbed it. rabbitui inverts the risk: it *starts* as the framework on qwertty. Keep the substrate contract as small as libvaxis's three-line event-loop contract so rabbitui itself remains one of several possible consumers of qwertty.
