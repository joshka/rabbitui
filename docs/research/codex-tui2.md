# Research memo: Codex CLI TUI and the tui2 rearchitecture

**Verdict:** tui2 is the most instructive *negative* result in the whole survey — a production coding-agent TUI (ratatui, Rust) that rearchitected its viewport so the app, not the terminal, owns scrollback + selection + copy, won decisively on resize-rewrap and copy fidelity, and was then **retired** because "fully native across the environment matrix creates a combinatorial explosion of edge cases." The lesson is not "own the viewport" or "don't"; it's that owning the viewport is a total commitment that trades a hard rendering problem for a harder *terminal-compatibility* problem, and a coding-agent CLI's users will punish you for the second one.

## The workload

Codex's TUI is a coding-agent chrome, and the survey's "flagship workload of the era" (recent-rust-tui-wave.md) is not hypothetical here — it's this crate. Concretely it must render, concurrently and under a live model stream:

- **A streaming markdown transcript.** Model output arrives as token deltas. `markdown_stream.rs` is a newline-gated collector: it buffers deltas and commits only *completed* lines at each newline (`commit_complete_source()`), re-rendering the accumulated prefix while the trailing partial line waits. So the transcript grows in commit ticks, not per-token — a deliberate throttle so markdown parsing and wrapping run on stable prefixes.
- **Structured, non-prose cells.** Diffs (`diff_render.rs`, a ~97 KB file in tui1), exec/tool-call output with exit status and timing (`exec_cell/`), reasoning summaries, session headers, MCP invocations. These are *not* markdown; they're typed cells with their own layout.
- **Approvals.** Command/patch approval requests are modal interactions that must interrupt the stream, take focus, and return a decision — handled in the bottom pane (`bottom_pane/approval_overlay.rs`) and as full-screen requests (`AppEvent::FullScreenApprovalRequest`).
- **A persistent bottom composer.** A multi-line textarea with file-search popups (`@`-mentions), slash-command popups, paste-burst handling, and a footer of key hints. It stays pinned at the bottom while the transcript scrolls above it.
- **Inline scrollback.** tui1's defining choice: finalized history lives in the terminal's *own* scrollback, not an alt-screen buffer. `insert_history.rs` writes committed cells above an inline viewport via raw escape sequences (scroll-region manipulation, `MoveTo`/`Print`), so `Ctrl+scroll`, native selection, and copy all "just work" because the terminal owns them. Alt screen is used only for overlays (transcript pager, diff view, onboarding).
- **Resumption.** Sessions persist and reload; the transcript must reconstruct from a rollout. (`resume_picker.rs` is 213 KB in tui1 — resumption is a large surface.)

### Where tui1's architecture hurt at scale

The pain is documented in `tui_viewport_and_history.md §1`. Because tui1 *cooperates with* terminal scrollback — inserting history above the viewport and occasionally clearing/rewriting regions — it inherited every terminal's idiosyncrasies:

- **Terminal-dependent breakage.** Scroll regions, clears, and resize semantics differ per emulator; "what looked correct in one terminal could drop or duplicate content in another." `insert_history.rs` already carries a `ZellijRaw` special-case and per-terminal branches.
- **Resize/reflow corruption.** History was printed to scrollback pre-wrapped at the width-of-the-moment. When the viewport moved or resized, tui1's attempt to keep in-memory history aligned with already-printed scrollback lines went out of sync — "some lines were lost or overwritten, others were duplicated." The line-count bookkeeping ("N lines printed before, skip N of the rewrapped transcript") was the specific bug generator.
- **"Clear and rewrite" doesn't save you.** Terminal.app leaves cleared screens as a scrollback page; iTerm2 gates full-scrollback clear behind user consent; terminals implement only subsets of the ANSI clear codes. Replaying a long session is expensive and races user output (shell prompts) outside alt screen.
- **No stable cell identity.** Once history is "dead text" in scrollback, you cannot address it — no per-cell copy, no expand/collapse of a tool call, no reflow — because you've handed those coordinates to the terminal.

The net, verbatim: tui1 "could not reliably guarantee 'the history you see on screen is complete, in order, and appears exactly once' across terminals, resizes, suspend/resume, and overlay transitions."

## tui2's design

tui2 shipped as a **separate crate** (`codex-tui2`), a fork of `codex-tui` behind a feature flag, so the revamp could roll out without breaking the shipping TUI (`#7793`, `#7833`; the dual-run glue is explicit in `Cargo.toml` — tui2 even `use`s `codex-tui` for shared primitives like `insert_history` and `custom_terminal`). The core moves:

- **The transcript owns the viewport; scrollback becomes an output target, not a data structure.** (`tui_viewport_and_history.md §2-3`.) The in-memory transcript — `transcript_cells: Vec<Arc<dyn HistoryCell>>` in `app.rs` — is the single source of truth. Every frame, tui2 flattens all cells into visual lines (remembering, per visual line, which cell + which line-within-cell it came from), picks a top line from a scroll anchor, clears the transcript region, and paints the visible slice. Alt screen is "a temporary render target… the only authoritative copy of the UI is the in-memory state."

- **State/widget identity = logical cells, not tree nodes.** The `HistoryCell` trait (`history_cell.rs`) is the unit of identity. A cell stores **width-agnostic content** (source string, diff, `Vec<MarkdownLogicalLine>`) and exposes `display_lines(width)` / `transcript_lines_with_joiners(width)` — it wraps *at render time*, per current width. Identity is a stable cell index, so selection, copy, and (future) per-cell interactions address cells by ID rather than fragile screen coordinates. This is not a retained widget tree à la Masonry/Xilem — it's a flat, append-only list of self-rendering logical cells with a live "active cell" that mutates in place while streaming (revisioned so caches invalidate).

- **Rendering/layout: a `Renderable` trait with intrinsic height, plus Flutter-style column/flex composition.** `render/renderable.rs` introduces `trait Renderable { fn render(&self, area, buf); fn desired_height(&self, width: u16) -> u16; fn cursor_pos(...) -> Option<(u16,u16)>; }` — note **`desired_height(width)`**, the content-measurement hook ratatui's 1-D constraint layout structurally lacks (see ratatui.md / discussion #552). On top of it: `ColumnRenderable` (stacks children by their desired height) and `FlexRenderable` (explicitly "loosely inspired by Flutter's Flex widget," with flex factors and last-child-gets-remainder rounding). This is a small, height-aware layout layer *above* ratatui's buffer, not a replacement for it — cells still emit `Vec<Line>` and paint into a `ratatui::Buffer`.

- **App-owned scrolling, selection, and copy.** Because tui2 owns the viewport, it must own mouse end-to-end (`tui_viewport_and_history.md §4`). Selection is expressed in **content-relative coordinates** over the flattened wrapped transcript (line index + column), so it tracks content as the transcript scrolls instead of gluing to a screen row; it excludes the left gutter; copy reconstructs the full range even off-screen, joins soft-wrapped prose into one logical line, and emits **markdown source** (backticks, fences) even where the UI renders it away. The soft-wrap/hard-break distinction is a first-class data structure: `TranscriptLinesWithJoiners { lines, joiner_before: Vec<Option<String>> }`, where `None` = hard newline and `Some(joiner)` = soft-wrap continuation carrying the exact skipped whitespace (`history_cell.rs`). This metadata "isn't needed in codex-tui" and exists only because copy is app-driven.

- **Scroll input is a measured, data-driven model.** `scroll_input_model.md` is remarkable: 16 probe logs, 13,734 events, 8 terminals. Finding: terminals emit 1–9+ raw events per physical wheel notch and *timing cannot distinguish wheel from trackpad*. The model groups events into **streams** (separated by 80 ms gaps or direction flips), **normalizes** by a per-terminal events-per-tick factor (AppleTerminal 3, Warp 9, WezTerm 1, iTerm 1, VS Code 1, Kitty 3…), classifies wheel-like (fixed 3 lines/tick, flush immediately) vs trackpad-like (fractional, carried across streams, 60 Hz cadence-gated, bounded acceleration), and exposes ~10 config knobs plus a `scroll_mode` override because "auto classification is a best-effort heuristic and must be overridable." Two prior approaches (cadence thresholds, pure count/duration classification) are documented as tried-and-rejected. This is what "own the viewport" actually costs: you reimplement the wheel.

- **Streaming reflow: parse markdown ≠ wrap markdown.** (`streaming_wrapping_design.md`.) The bug: streaming baked soft wraps as `Vec<Line>` into cells, so a wider viewport couldn't un-split them — ratatui's `Vec<Line>` can't distinguish a soft wrap from a newline. The fix: stream a width-agnostic `MarkdownLogicalLine { content, initial_indent, subsequent_indent, line_style, is_preformatted }`; `AgentMessageCell` stores logical lines and wraps at render time; a `TranscriptViewCache` keyed on width rebuilds on resize. Rule of thumb they extracted: *any cell that stores pre-wrapped `Vec<Line>` for prose is a reflow bug waiting to happen; store source/logical lines and wrap inside `display_lines(width)`.*

- **Event loop / programming model: actor-style tokio, coalescing frame requests, message-passing.** `frame_requester.rs` is an actor (per Alice Ryhl's "Actors with Tokio"): a `FrameRequester` handle any task can clone to `schedule_frame()` / `schedule_frame_in(dur)`; a `FrameScheduler` task coalesces requests and clamps draws to 60 FPS via `FrameRateLimiter`, notifying the loop on a broadcast channel. Redraws are *requested*, not run per-input. State updates flow as an `AppEvent` enum (`app_event.rs`) — `CodexEvent`, `InsertHistoryCell(Box<dyn HistoryCell>)`, `CommitTick`, approvals, model/config updates. **Note this message model is shared with tui1** (tui1 also has a ~1100-line `AppEvent` enum and the same `bottom_pane` structure) — the event loop was *not* what tui2 changed. The novelty is entirely in the render/viewport/transcript layer.

- **Testing story: a real VT100 emulator in the loop.** `test_backend.rs` wraps `CrosstermBackend<vt100::Parser>` so tests drive the *actual* escape-sequence output through a terminal emulator and assert on the resulting screen grid (`tests/suite/vt100_history.rs`: insert two lines, assert the emulated screen contains them; wrap a 45-char token at width 20, count the 'A's on screen). This is stronger than ratatui's `TestBackend` buffer-equality because it validates the ANSI *emission*, not just the intended buffer — exactly what you need when your correctness bugs live in scroll-region escapes. Plus snapshot tests (normalized in `#7833`) and resize-reflow tests added with the streaming fix (`agent_message_cell_reflows_streamed_prose_on_resize`).

## What it deliberately avoided

- **It did not build a retained widget tree.** No node graph, no `on_event/layout/paint/children`. Identity is a flat list of logical cells; layout is intrinsic-height column/flex, not a 2-D constraint tree. The "interactive cells" ambitions (expand/collapse a tool call, per-cell copy, drill-into overlay) in the roadmap are framed as *unlocked by* cell identity, not as a tree rewrite.
- **It did not reflow already-printed scrollback.** Explicit non-goal (`streaming_wrapping_design.md`, `tui_viewport_and_history.md §6.3`): scrollback is **append-only**, correctness is "each logical cell appears exactly once," not pixel-perfect layout. Reflowing old scrollback runs straight back into the per-terminal clear/scroll-region swamp they were fleeing.
- **It did not fully commit to alt-screen.** tui2 keeps history in normal scrollback on suspend/exit (a **cell-level high-water mark** counts logical cells already printed, so width changes never cause drop/dup — the fix for tui1's line-count bug). It uses alt screen only for the interactive session and overlays. It deliberately straddles inline and alt-screen rather than going full-alt-screen (which would have sidestepped the native-terminal fight but abandoned inline scrollback — the thing agent-CLI users prize).
- **It did not ship suspend-time scrollback printing, streaming display-time wrapping everywhere, multi-click selection, or find-in-transcript at the pinned commit** — all P1/P2 roadmap, several never landed before retirement.
- **It did not try to auto-detect wheel vs trackpad from device info** (unavailable) or timing (proven non-separable). It shipped a heuristic + mandatory user override instead.
- **And ultimately, OpenAI did not keep it.** The retirement commit (`#9640`) is the honest verdict, worth quoting in full because it is this project's own author writing it: *"What worked: a transcript-owned viewport delivered excellent resize rewrap and high-fidelity copy (especially for code). Why stop: making that experience feel fully native across the environment matrix (terminal emulator, OS, input modality, multiplexer, font/theme, alt-screen behavior) creates a combinatorial explosion of edge cases. What next: redraw-based improvements to the existing terminal-native TUI so scrolling, selection, and copy remain native while resize/redraw correctness improves."* Issue #8344 was the leading edge of that constituency. tui2 solved the rendering problem and lost the compatibility problem, and for a CLI shipping to every terminal on earth, the second one is the product.

## Implications for rabbitui

- **Ship intrinsic-height layout (`desired_height(width)`) from v0.1, because tui2 had to bolt it onto ratatui to do anything real.** The single most reused new primitive in tui2 is `Renderable::desired_height(width)` + `Column`/`Flex` composition — the exact content-measurement gap ratatui.md and discussion #552 flag. rabbitui's taffy-style measure-function layout (already in ratatui.md's implications) is directly validated by a production app reinventing a slice of it.
- **Make the transcript/scrollback model a *studied* design axis with two supported targets, because tui2 proves inline-viewport ownership is a cliff, not a slope.** Do not default to "own the viewport." Offer (a) a terminal-native inline mode where the terminal keeps scrollback/selection/copy, and (b) an app-owned viewport mode — and document, loudly, that mode (b) obligates you to reimplement wheel physics, selection geometry, copy fidelity, and per-terminal escape quirks. tui2 chose (b) and it was the right *engineering* and wrong *product* call. Give app authors the choice with eyes open.
- **Store source, wrap at render time; never persist soft wraps as `Vec<Line>` for reflowable prose.** Bake tui2's hard-won rule into the widget contract: reflowable content stores width-agnostic logical lines (with `initial_indent`/`subsequent_indent`/`is_preformatted`), and wrapping happens in the paint/measure path keyed on width. Preformatted/code cells opt out. This is a correctness requirement for any streaming-markdown workload and a subtle bug generator if left to convention.
- **Model soft-wrap-vs-hard-break as data, not heuristic, so copy is faithful.** Adopt tui2's `joiner_before: Vec<Option<String>>` idea: the wrapping algorithm that *creates* a break is the only thing that knows whether it was soft, and what whitespace it skipped. Any framework offering app-driven selection/copy over wrapped text needs this metadata plumbed from wrap to clipboard, or it will guess (and copy code wrong).
- **Put a VT100-emulator backend in the test kit, above buffer-equality.** ratatui.md already argues for a headless driver; tui2 shows the specific missing rung: `CrosstermBackend<vt100::Parser>` asserting on the *emulated screen* validates escape-sequence emission (scroll regions, clears, cursor moves) that a `TestBackend` buffer comparison cannot. Inline mode, resize-reflow, and suspend/resume are exactly the paths whose bugs only appear at the ANSI level. Ship this as a first-class harness.
- **Own frame scheduling with a coalescing, rate-limited requester — the one piece of tui2's runtime worth copying wholesale.** `FrameRequester`/`FrameScheduler` (clone-a-handle, `schedule_frame_in(dur)`, coalesce to 60 FPS on a broadcast channel) is the clean actor pattern for "redraw is requested by many tasks, run at most 60x/s." This is orthogonal to the viewport debate and validates ratatui.md's "own the event loop, async-first" implication with a concrete shape. Note the message-passing `AppEvent` loop was *not* tui2's differentiator — both tui1 and tui2 have it — so don't over-index on Elm-style messaging as the hard part; the hard part is layout, wrapping, and terminal compatibility.
- **Treat scroll-wheel physics as a real, terminal-specific subsystem, and always expose an override.** If rabbitui ever owns scrolling, `scroll_input_model.md` is the spec: per-terminal events-per-tick normalization, stream grouping, wheel-vs-trackpad heuristic, and a mandatory `scroll_mode` config escape hatch. Ship the probe-derived defaults table. This is the least glamorous and most under-budgeted cost of viewport ownership.
- **Say the quiet part in the docs: "native terminal behavior" is a first-class feature users will defend.** Issue #8344 and the tui2 retirement are the same lesson from both ends. rabbitui's inline-mode design (recent-rust-tui-wave.md's top implication) should be explicit that preserving native scrolling/selection/copy is often *more* valuable than owning them — and that the app-owned path is justified only when the workload (rich per-cell interaction, faithful code copy across reflow) genuinely demands it. Design the seam so a team can start native and escalate to owned per-feature, not all-or-nothing as tui2 was forced to.
