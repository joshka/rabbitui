# Rust GUI Lessons for a Terminal Framework

**Verdict:** Steal the Rust GUI world's answers to tree ownership, widget identity, and incrementality wholesale — but refuse its rendering, compositing, and text-shaping machinery, because a cell grid with cheap full redraws dissolves the problems that machinery exists to solve.

## The problems

Ten years of Rust GUI work reduces to five problems. Only some of them are about pixels.

1. **Who owns the tree.** A retained OO widget tree is "a big wad of shared mutable state" (Rust 2020 post): containers own children while callbacks, focus managers, and layout also need references to them. `Rc<RefCell<T>>` webs are the escape hatch and are universally judged unidiomatic and panic-prone (ECS-UI post; Warp post: "it's hard to share data across this component tree without inheritance"). Every viable Rust architecture — ECS, lenses+diffing, Crochet, Xilem, Masonry's arena, Dioxus's generational-box — is a different way to route exactly one `&mut` through the tree.

2. **Widget identity.** Focus, accessibility, event routing, animation, and incremental update all need a widget to be "the same widget" across frames. Object references are messy in Rust; the survivors are key/id **paths** (Xilem, Compose) and **arena keys** (Masonry `tree_arena`, Dioxus, Warp's EntityId HashMap). Pure immediate mode "fundamentally struggles" here (Principled reactive UI) and accessibility APIs, which assume a retained stably-identified tree, "eliminate pure immediate mode as viable for production" (Next dozen GUIs).

3. **Incrementality.** "Hidden inside of every UI framework is some kind of incrementalization framework" (Xilem post, quoting Minsky). Elm/Iced-style rebuild-the-whole-view-every-16ms "does not satisfy the actually incremental goal" (Principled reactive UI); full incremental engines (SwiftUI ViewGraph, Adapton) are judged infeasible to build in Rust without enormous resources. The livable middle: diffing with memoize nodes, `Arc::ptr_eq` shortcuts, waker-style narrow change notifications, and diff-and-patch mutation objects containers can skip when empty.

4. **State scoping.** Widgets written against a small state type must embed in apps with a big one. Druid's lenses (CPS projections `with`/`with_mut`, Druid book) did this but forced the whole app state into one unified, clonable, diffable tree with no interior mutability — maintainers were "unsatisfied with the crate's API and main abstractions for years" and it killed Druid (Druid ecosystem post). Xilem's Adapt (a closure over `&mut ParentState`) and Dioxus's Copy signals are the looser successors.

5. **Text and rendering depth.** Text is "a litmus test for how far along a UI toolkit has gotten" (Next dozen GUIs): shaping, itemization, BiDi, IME, fallback, accessibility. Rendering demands a GPU strategy, a compositor relationship, damage regions, and layered scenes. These consumed the majority of Linebender's calendar time (Xilem 2024 post-mortem: foundational yak-shaving on text, compositing, accessibility was unavoidable).

## What GUI frameworks concluded

- **View/element split wins.** Xilem's three trees — ephemeral statically-typed view tree rebuilt each cycle, long-lived type-erased widget tree, persistent view-state tree — give a declarative surface over a retained incremental core (Xilem post). Linebender then promoted the split to an ecosystem boundary: Masonry is an unopinionated retained backend under swappable reactive frontends, and porting Xilem onto it eliminated whole classes of cruft (Masonry 0.1; 2024 post-mortem). The reactive layer and the widget engine are **separable products** (Raph 2023).
- **Identity is data: id paths or arena keys.** Xilem's id paths ([1,3] from root) triple as identity, event route, and async waker address. Masonry's biggest 2024 win was moving from parent-owned children to arena/slotmap storage where "widgets own keys into a shared structure," solving "mutating an item's children while keeping a live reference to the item's value" (post-mortem).
- **Traversal belongs to the framework, not to widgets.** Masonry's standardized passes (event → rewrite: mutate/update/layout/compose → render) with capability-typed contexts made pass code "more concise, clean and readable than it ever was," and let cross-cutting behaviors (tab focus, pointer icons) be implemented once in passes instead of in every widget (post-mortem; pass-system docs). Layout (expensive, ripples to siblings) is split from compose (cheap transform-only, subtree-local) — that's how scrolling avoids relayout.
- **Events thread one `&mut`.** Xilem dispatches down the id path, threading a single mutable borrow of app state to the leaf callback: no observers, no `setState`, no interior mutability. Child→parent communication is typed return values, not a global message enum (Xilem post).
- **Failed experiments are load-bearing.** Crochet (imgui API over retained core) died of state tearing, no optional/named args, unenforced nesting, and a single-threaded mutable-context bottleneck (Next dozen GUIs). Dioxus threw away its bump-arena lifetimes because "newcomers are easily scared off by lifetime issues" and rebuilt on Copy signals over generational arenas (Dioxus 0.5). xi-editor's async front-end/core split: "adding async made everything more complicated, in some cases considerably so" — keep the core synchronous (xi retrospective). Druid went ECS → lenses → Xilem: **expect to iterate the core architecture**.
- **Floem is the counterpoint worth knowing:** build the view tree once, update retained views in place via fine-grained signals — no diffing at all, and no reconciler identity problem, at the cost of explicit constructs (`dyn_view`, `VirtualList`) for dynamic structure (Floem README). It outsources layout to Taffy.
- **Tooling as architecture.** Masonry's "fearless GUI" pillars — headless TestHarness, snapshot tests, `children()` reflection, planned input record/replay — only work because the framework owns storage and traversal (Masonry 0.1).

## What transfers to terminals

These problems arrive in a TUI **undiminished**, because they are Rust problems and reactivity problems, not pixel problems:

- **Tree ownership.** A ratatui-style stateless render dodges this only until you want retained widgets, focus, and incremental update. The moment rabbitui retains anything, it inherits the full borrow-checker fight and should adopt the known answer (arena + keys) on day one, not discover it in year two like Masonry did.
- **Widget identity and focus.** Tab traversal, scroll-position persistence, cursor/IME position, and any future screen-reader story all require stable identity across frames. (Caveat: an AccessKit-style terminal export is plausible but unbuilt — Levien hedges even for GUI: "you never know for sure until you actually try it" (Xilem post), and terminal AccessKit is an open ratatui discussion, #2190, not a fact.) Stable identity is *the* thing ratatui's immediate-mode model cannot give you, and the GUI literature is unanimous that it must be structural-position-plus-explicit-keys, carried as paths or arena keys — though prior-art.md shows the labs carry id-paths *without* a retained tree.
- **State ownership & scoping.** "Who owns app state and how does a child widget see a slice of it" is identical in a TUI. Lenses are a proven dead end; Adapt-style closures over `&mut` and single-borrow event threading transfer directly.
- **Reactivity/incrementality — but for the *view* computation, not the paint.** Even with cheap redraws, re-running user view code (building lists, formatting text, running layout) every frame at 60fps on large models is real cost. Xilem's lightweight kit (memoize nodes, ptr-eq, skip-empty-delta) transfers; a full incremental engine does not and never did.
- **Pass systems.** Event → rewrite → render with typed contexts, bounded rework loops, and a compose (offset-only) pass separate from layout maps beautifully onto a cell grid: scrolling a viewport is a compose-pass concern, not a relayout.
- **Line breaking and segmentation.** The lower half of the text stack survives shaping's removal: grapheme clusters, Unicode width (East Asian ambiguity), the line-break candidate ladder (word boundaries → soft hyphens → graphemes), and BiDi run analysis if you care about RTL. "Much of the knowledge is arcane, and there is unfortunately no good textbook" (Minikin retrospective) applies verbatim to terminal width handling.
- **Async discipline.** Xilem's design (view nodes hold futures, id paths as wakers, reuse tokio) and xi's warning (don't put an async boundary between core and render loop) jointly define rabbitui's lane: async-first at the edges, synchronous single-threaded update/layout/paint core.
- **Testability & replayability.** Headless harness + buffer snapshots + input record/replay are *easier* in a terminal (the frame is a small grid of cells) — this is the one place rabbitui can exceed the GUI state of the art cheaply.

## What terminals dodge

Be explicit about the machinery rabbitui must **not** import:

- **The GPU/compositor axis — entirely.** Vello, wgpu, scene graphs, layer trees, the "compositor relationship" decision (Next dozen GUIs), retained draw trees to minimize GPU upload. A frame is ≤ ~100k cells; diffing two buffers and emitting escape sequences is microseconds. Consequence: **damage regions are unnecessary** — the double-buffer diff *is* the damage tracking, computed for free after the fact. Druid's incremental painting via damage regions (Rust 2021) solved a cost class terminals don't have. Keep the paint model "everyone writes into a fresh buffer; framework diffs."
- **Most of the text stack.** No shaping (HarfBuzz/Swash), no itemization/font-fallback (the "underspecified swamp" of the text-layout post), no subpixel anything, no font queries as a bottleneck (Minikin). The terminal emulator (qwertty) owns glyphs. rabbitui keeps only segmentation, width, and line breaking. Caveat: width is *not* trivially compositional — like shaping, a grapheme's cells can depend on terminal mode and context (mode-2027, emoji VS16), so centralize width policy in one crate-level oracle rather than scattering `unicode-width` calls.
- **Sub-frame incremental paint.** React's virtual DOM exists to amortize an expensive JS↔DOM boundary (Unified theory); Flutter's repaint boundaries exist because rasterization is expensive. A TUI has one cheap in-process sink. Do not build repaint-boundary machinery; spend the complexity budget on skipping *view/layout* work instead.
- **Windowing/platform fragmentation.** No winit-vs-tao, no menus/IME/native-dialog impedance (Next dozen GUIs' windowing lament), no per-OS text APIs, no "native look" debate — Levien already declared "native" a meaningless axis. One window, one grid, capabilities negotiated via escape sequences (kitty keyboard, synchronized output — the terminal analog of the compositor decision, and a far smaller one).
- **Monomorphization/binary-size pressure — mostly.** SwiftUI-grade type gymnastics were partly about perf on huge trees; TUI trees are small. Still use Xilem's AnyView erasure trick, but for ergonomics, not size.
- **Layout at GUI fidelity.** Full CSS grid/flexbox generality, fractional pixels, rounding/hinting. Integer cell arithmetic makes layout radically simpler; taffy-style flexbox over `u16` cells is plenty, and cassowary (ratatui's inheritance) is over-general for the need.

## Implications for rabbitui

- **Programming model: this memo's GUI evidence favors the Xilem view/element split over Elm, React-in-Rust, and pure immediate mode — but that is a recommendation from GUI-domain evidence only, and the sibling memos reach different conclusions from terminal-domain evidence. Treat this as an open fork, not a decision.** The shape: ephemeral typed view tree diffed each cycle against a retained widget tree, single `&mut AppState` threaded down id paths to callbacks. Sorting the rationale by how well it was verified to transfer to cell grids:
  - *Transfers, with terminal corroboration:* Elm's message-enum ergonomics are documented losers in GUI (Xilem post; Rust 2021 concedes Elm is *easier* but not incremental) **and** in the terminal (tui-realm's `Msg`/`Cmd` double bookkeeping is "tolerated, not loved" — prior-art.md). The identity requirement is real in terminals too (focus, scroll persistence, IME — see "What transfers"), though prior-art.md shows id-paths can be carried without a retained tree.
  - *Assumed, not verified:* the accessibility argument against immediate mode ("Next dozen GUIs") presupposes an AccessKit-style terminal export that nobody has built (ratatui #2190 is exploratory; Levien himself hedges). The incremental-view-cost argument is calibrated to GUI-scale trees; TUI trees are small (see the monomorphization bullet), and textual.md shows whole-subtree recompose working as an app-scale opt-in. Note also: Levien's actual words are that trying the architecture "in other domains" — TUI among them — would test it; he did not claim it fits.
  - *The fork the orchestrator must resolve:* textual.md concludes retained tree + reactive attributes is the model *proven at terminal app scale*, with Xilem-style diffing as a per-widget opt-in; prior-art.md concludes the labs' frame-facts model should be the core, with an Xilem-style view-diff layer as an optional crate above it, never the widget contract. This memo's evidence rules out a pure Elm monolith and establishes that any choice needs stable identity plus a home for retained state — it cannot adjudicate among the three survivors.
  - *Floem/signals: not actually rejected on terminal evidence.* The GUI objection (dynamic structure pushed onto users via `dyn_view`/`VirtualList`) stands, but the same build-once-mutate-via-reactivity shape is Textual's model — the best-attested at terminal app scale (textual.md) — while its Rust-TUI instance, rooibos, stalled on adoption, not architecture (prior-art.md). If the fork resolves toward a retained core, signals re-enter as a live frontend candidate.
- **Split rabbitui-core (retained widget arena + passes) from rabbitui-view (reactive layer) as separate crates from day one**, because Masonry⇄Xilem proved the boundary is real and it lets ratatui-style imperative users target core directly. But heed xi's warning: keep it two crates in one process/loop, never an async protocol boundary.
- **Widget storage: arena/slotmap keyed by generational ids; containers hold keys, never children.** This is Masonry's hardest-won lesson (2024 post-mortem) and it also buys the inspector, serialization, and headless testing for free. Widget identity exposed to users as id paths (structural + explicit keys for reordered lists), which double as focus addresses, event routes, and async waker payloads.
- **Rendering: keep ratatui's double-buffer diff; add z-layers as N buffers composited at diff time; skip damage regions entirely.** Cheap full paint means correctness-first painting with zero invalidation bugs. Put the incrementality budget in view-land: `Memoize` nodes + `Arc::ptr_eq` + skip-empty-delta reconciliation.
- **Passes: specify the Masonry pipeline in cell units** — event (targeted + bubble) → mutate → update_tree → update_focus → layout → **compose (integer offsets; scrolling lives here)** → paint, with capability-typed contexts (PaintCtx can't request layout) and a bounded rework loop. Framework-owned traversal; the third-party Widget trait implements per-node behavior, never recursion. Keep `children()` reflection mandatory for tooling.
- **Layout: taffy (flexbox/grid) over u16 cells, not cassowary.** Floem shows taffy works under a reactive tree; constraint solvers are the over-imported GUI machinery here. Text measurement goes through one width oracle crate shared with qwertty so framework and emulator agree on grapheme widths (mode-2027-aware).
- **Concurrency: async-first shell, synchronous core.** Event loop owned by rabbitui on qwertty's async substrate; timers/subprocess/network integrate as futures held by view nodes with id-path wakers (Xilem's design); update/layout/paint runs synchronously in one thread. xi-editor is the standing proof that an async core boundary multiplies complexity for nothing.
- **Input/focus: implement tab focus, focus rings, and bubbling as framework passes, not widget code** (Masonry's exact win). Kitty-keyboard/enhanced-input capability negotiation belongs in qwertty; rabbitui consumes a normalized event type. Design the retained tree so an AccessKit-style export is possible later — it costs nothing now and is impossible to retrofit onto immediate mode.
- **Styling: this is the known unsolved hole — plan it before code.** Masonry named "efficiently associate values of arbitrary types with any widget" (the ECS Component slot) as its single biggest usability blocker. For a Textual-CSS-equivalent, that means: style properties as typed side-tables keyed by widget id in the arena, resolved in an update pass — decided in the storage design, not bolted on.
- **Widget trait: public, minimal, framework-traversed.** Masonry's "users shouldn't write custom widgets" proposal drew "strong negative reactions" — third-party widgets are the ecosystem (ratatui proves it). Follow the Vello usage survey for API shape: tiered methods with minimal-arg defaults (`fill()` / `fill_with()`), because clients use a tiny subset through middleware.
- **Testing: make the headless harness a launch feature, not an afterthought.** Buffer snapshot tests, simulated input, tree assertions via reflection, and input record/replay — the "fearless GUI" agenda is fully achievable in a terminal and is a genuine differentiator over both ratatui and GUI toolkits.
- **Process: budget for one architecture rewrite and validate against one real demanding app** (Runebender discipline; Druid's ECS→lens→Xilem arc; Faure: "design processes need weeks or months of background reflection"). Ship the arena + pass core first; treat the reactive layer as replaceable.
